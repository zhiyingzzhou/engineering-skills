# Test Strategy

本页规则以 Testing Library 的用户视角原则和 Playwright 的关键路径实践为基线。

## 分层原则

- 逻辑规则优先在最便宜、最快的层级验证
- 组件交互优先验证可见 DOM 和用户行为
- 跨页面、跨系统关键流程用少量 E2E 兜底
- 不要把所有问题都推给最慢的 E2E

## 选择标准

- 如果风险在纯计算或状态变换，选单元层
- 如果风险在渲染、可达性和交互反馈，选 DOM 层
- 如果风险在路由、网络、权限或真实浏览器行为，选 E2E 层

## 编写原则

- 一个测试只保护一个清晰行为
- 测试命名应反映用户意图和预期结果
- 只 mock 真正不属于本层责任的边界
